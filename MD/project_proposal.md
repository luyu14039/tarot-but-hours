# “司辰塔罗” (Tarot of the Hours) 互动项目策划方案

## 1. 项目概述与评估

**项目评价：**
这是一个**极具潜力且契合度极高**的二创方向。《密教模拟器》与《司辰之书》的核心魅力在于其晦涩、神秘且充满隐喻的文本风格，这与塔罗占卜的模糊性、象征性不谋而合。

*   **优势**：
    *   **世界观契合**：司辰本身就代表了宇宙的某种法则（准则），非常适合作为原型的象征。
    *   **粉丝粘性**：该圈层粉丝热衷于解读文本（Lore-diving），LLM 生成的谜语人风格回答会非常受欢迎。
    *   **扩展性**：40张牌的设定打破了传统塔罗的大阿卡纳限制，构建了一套独特的占卜体系。

*   **挑战**：
    *   **LLM 调优**：如何让 AI 既能准确回答用户问题，又能保持“密教”那股疯癫、诗意、克苏鲁式的文风（“无形之术”的风格）。
    *   **美术资源**：需要统一且高质量的卡面视觉风格（可以使用游戏原画或二创风格化处理）。

## 2. 核心功能设计

### 2.1 牌阵系统 (The Spreads)
不使用通用的塔罗牌阵名称，而是进行“密教化”包装：
*   **单张牌 - “林地的一瞥” (Glimpse of the Wood)**：简单的每日运势或是非题。
*   **三张牌 - “漫宿的道路” (Path of the Mansus)**：
    *   *过去 (雄鹿之门)* / *现在 (蜘蛛之门)* / *未来 (孔雀之门)*。
*   **五张牌 - “十字路的故事” (The Crossroad)**：针对具体问题的深度解析。

### 2.2 40位司辰的映射逻辑
您提到的 0-XXXIX (0-39) 结构是一个独特的体系。建议将牌面分为几个层级，以便 LLM 理解权重：
*   **0-XXI (0-21)**：对应传统大阿卡纳，作为核心解答。
*   **XXII-XXXIX (22-39)**：包含虚空诸神、已逝司辰等。这些牌出现时，可以代表“隐秘的影响”、“不可控的变量”或“来自过去的阴影”。

### 2.3 LLM “守密人” 人格
AI 不应只是一个冷冰冰的占卜师，而应该扮演一个角色：
*   **风格 A：夜游者** - 语气疲惫，充满警告，仿佛在梦境边缘低语。
*   **风格 B：噤声局探员** - 语气公事公办，将占卜视为“对现实扭曲的评估”，并警告用户不要深究。
*   **风格 C：破碎的预言家** - 说话断断续续，大量引用游戏内的书籍原文（如《战栗之诗》）。

## 3. 技术架构方案选择

为了满足您的需求，我提供三种不同侧重点的方案供选择：

### 方案一：轻量级 Web 应用 (推荐)
**侧重：** 快速开发，易于分享，成本低。
*   **前端**：React (Next.js) 或 Vue (Nuxt.js)。
    *   使用 Tailwind CSS 实现极简的黑金/暗红配色风格。
    *   卡牌翻转动画效果。
*   **后端**：Next.js API Routes (Serverless)。
*   **AI 接入**：OpenAI API (GPT-4o) 或 DeepSeek (性价比高，中文理解好)。
*   **部署**：Vercel (免费且极速)。
*   **优点**：用户体验极佳，无需维护服务器，链接即开即用。

### 方案二：Python 全栈 (Streamlit/Gradio)
**侧重：** 逻辑开发极快，适合验证原型，无需太多前端知识。
*   **框架**：Streamlit。
*   **逻辑**：
    *   用户输入 -> Python 脚本处理 -> 调用 LLM -> 渲染 Markdown 结果。
*   **优点**：开发速度最快，Python 处理文本和逻辑非常方便。
*   **缺点**：界面定制能力较弱，很难做出那种“神秘仪式感”的动画效果。

### 方案三：沉浸式 3D/2D 体验 (Three.js / Godot Web)
**侧重：** 氛围感，视觉冲击，游戏化体验。
*   **技术**：Three.js (Web 3D) 或 Godot 引擎导出 HTML5。
*   **体验**：
    *   用户看到一张 3D 的桌子，上面摆着蜡烛和卡牌。
    *   鼠标拖拽抽牌，物理效果。
*   **优点**：最符合“密教模拟器”的游戏体验，代入感最强。
*   **缺点**：开发周期长，技术门槛高。

## 4. 详细实施路线图 (以方案一为例)

### 第一阶段：原型 (MVP)
1.  **数据准备**：完善 `list.md`，为每位司辰添加“正位/逆位”关键词（或“飞升/沉沦”含义）。
2.  **基础页面**：输入框 + 抽牌按钮 + 随机显示 3 张卡牌图片。
3.  **Prompt 工程**：调试 System Prompt，让 AI 学会“说人话但又不说人话”。
    *   *Prompt 示例*：“你是一位精通无形之术的学者。用户抽到了‘飞蛾’。请结合‘混乱、剪除旧皮、寻找光’的意象，解答用户关于‘是否应该辞职’的问题。回答要晦涩但最终给出指引。”

### 第二阶段：视觉与交互
1.  **卡面设计**：收集或生成 40 张卡牌图（统一边框、牌背）。
2.  **仪式感**：添加抽牌时的音效（翻书声、低语声）和动效（淡入淡出、粒子效果）。

### 第三阶段：社区与分享
1.  **生成分享图**：将占卜结果生成一张带有神秘符号的图片，方便用户发朋友圈/微博。
2.  **历史记录**：保存用户的占卜记录（“你的密教档案”）。

## 5. 待确认事项
*   **图片版权**：是否使用官方素材（需遵循 Weather Factory 的二创规定）还是使用 AI 生成/手绘？
*   **API 成本**：是您提供 Key 供大家免费使用（需设置限额），还是用户自己填 Key？

建议从 **方案一 (Next.js)** 开始，因为它在表现力和开发难度之间取得了最好的平衡。
